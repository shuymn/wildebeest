# Wildebeest TypeScript Type Definitions

This document defines TypeScript types that correspond to the database schema and Mastodon API entities.

## Design Principles

1. **Three-Layer Type System**:
   - **Database Row Types**: Direct mappings to SQLite columns (snake_case)
   - **Domain Model Types**: Internal application types (camelCase)
   - **API Response Types**: Mastodon API compatible response shapes

2. **Strict Nullability**: Use `null` for nullable database columns, `undefined` for optional API fields

3. **Branded Types**: Use branded types for IDs to prevent accidental mixing

---

## Branded ID Types

```typescript
// Prevents accidental mixing of different ID types
declare const brand: unique symbol

export type Brand<T, B> = T & { [brand]: B }

// Entity ID types
export type AccountId = Brand<string, 'AccountId'>
export type StatusId = Brand<string, 'StatusId'>
export type MediaAttachmentId = Brand<string, 'MediaAttachmentId'>
export type PollId = Brand<string, 'PollId'>
export type NotificationId = Brand<string, 'NotificationId'>
export type ListId = Brand<string, 'ListId'>
export type FilterId = Brand<string, 'FilterId'>
export type ReportId = Brand<string, 'ReportId'>
export type ConversationId = Brand<string, 'ConversationId'>
export type TagId = Brand<number, 'TagId'>
export type CustomEmojiId = Brand<number, 'CustomEmojiId'>

// Mastodon-style IDs (for API compatibility)
export type MastodonId = Brand<string, 'MastodonId'>
```

### Retention Config

```typescript
/**
 * Configuration for data retention policies.
 * All duration values are in days.
 * null means "never expire".
 */
export interface RetentionConfig {
  remoteStatuses: {
    /** Default TTL for remote statuses (default: 30) */
    defaultTtlDays: number
    /** TTL for remote statuses with local interactions (default: 90) */
    interactedTtlDays: number
    /** TTL for statuses in reply chains with local statuses (default: 60) */
    replyChainTtlDays: number
  }
  remoteAccounts: {
    /** Default TTL for remote accounts (default: 90) */
    defaultTtlDays: number
    /** TTL for followed remote accounts (default: null = never) */
    followedTtlDays: number | null
  }
  notifications: {
    /** TTL for notifications (default: 90) */
    ttlDays: number
  }
  inboxEntries: {
    /** TTL for processed inbox entries (default: 1) */
    processedTtlDays: number
  }
  deliveryFailures: {
    /** TTL for delivery failure records (default: 7) */
    ttlDays: number
  }
}

/**
 * Environment variables for retention configuration.
 */
export interface RetentionEnvVars {
  RETENTION_REMOTE_STATUSES_DAYS?: string
  RETENTION_REMOTE_STATUSES_INTERACTED_DAYS?: string
  RETENTION_REMOTE_STATUSES_REPLY_CHAIN_DAYS?: string
  RETENTION_REMOTE_ACCOUNTS_DAYS?: string
  RETENTION_REMOTE_ACCOUNTS_FOLLOWED_DAYS?: string
  RETENTION_NOTIFICATIONS_DAYS?: string
  RETENTION_INBOX_PROCESSED_DAYS?: string
  RETENTION_DELIVERY_FAILURES_DAYS?: string
}

/**
 * Default retention configuration values.
 */
export const DEFAULT_RETENTION_CONFIG: RetentionConfig = {
  remoteStatuses: {
    defaultTtlDays: 30,
    interactedTtlDays: 90,
    replyChainTtlDays: 60
  },
  remoteAccounts: {
    defaultTtlDays: 90,
    followedTtlDays: null  // Never expire if followed
  },
  notifications: {
    ttlDays: 90
  },
  inboxEntries: {
    processedTtlDays: 1
  },
  deliveryFailures: {
    ttlDays: 7
  }
}

/**
 * Parse retention config from environment variables.
 */
export function parseRetentionConfig(env: RetentionEnvVars): RetentionConfig {
  const parseNumber = (value: string | undefined, defaultValue: number): number => {
    if (value === undefined) return defaultValue
    const parsed = parseInt(value, 10)
    return isNaN(parsed) ? defaultValue : parsed
  }

  const parseNullableNumber = (value: string | undefined, defaultValue: number | null): number | null => {
    if (value === undefined) return defaultValue
    if (value.toLowerCase() === 'never' || value === '') return null
    const parsed = parseInt(value, 10)
    return isNaN(parsed) ? defaultValue : parsed
  }

  return {
    remoteStatuses: {
      defaultTtlDays: parseNumber(env.RETENTION_REMOTE_STATUSES_DAYS, 30),
      interactedTtlDays: parseNumber(env.RETENTION_REMOTE_STATUSES_INTERACTED_DAYS, 90),
      replyChainTtlDays: parseNumber(env.RETENTION_REMOTE_STATUSES_REPLY_CHAIN_DAYS, 60)
    },
    remoteAccounts: {
      defaultTtlDays: parseNumber(env.RETENTION_REMOTE_ACCOUNTS_DAYS, 90),
      followedTtlDays: parseNullableNumber(env.RETENTION_REMOTE_ACCOUNTS_FOLLOWED_DAYS, null)
    },
    notifications: {
      ttlDays: parseNumber(env.RETENTION_NOTIFICATIONS_DAYS, 90)
    },
    inboxEntries: {
      processedTtlDays: parseNumber(env.RETENTION_INBOX_PROCESSED_DAYS, 1)
    },
    deliveryFailures: {
      ttlDays: parseNumber(env.RETENTION_DELIVERY_FAILURES_DAYS, 7)
    }
  }
}
```

## Summary

This type system provides:

1. **Type Safety**: Branded IDs prevent mixing different entity types
2. **Database Mapping**: Row types directly correspond to SQLite schema
3. **Domain Logic**: Internal types with proper TypeScript primitives
4. **API Compatibility**: Response types match Mastodon API exactly
5. **Validation**: JSON schemas for flexible properties columns
6. **Utilities**: Pagination, type guards, and converters

The three-layer approach allows:
- Efficient database queries with row types
- Clean business logic with domain types
- Compatible API responses for clients
